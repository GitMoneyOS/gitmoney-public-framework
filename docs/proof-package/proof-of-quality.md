# Proof Of Quality

## Purpose

Define how GitMoneyOS judges the quality of its own work.

The quality bar is not "more docs." The quality bar is governed evidence that a founder can inspect.

## Quality Standard

Every meaningful GitMoney artifact should answer:

- What is the business reason?
- What layer does this belong to: memory, agent workbench, GitHub record, or cross-layer?
- Who owns it?
- What is the approval path?
- What evidence proves it?
- What remains uncertain?
- What is the next checkpoint?

## Approval Packet Truth Test

A pull request is only a good approval packet if a founder can answer:

- What changed?
- Why now?
- Which files or categories changed?
- What proof exists?
- What decision is needed?
- What happens after merge?
- What risk remains?

If the PR body understates the real changed-file surface, the packet is incomplete.

## Self-Healing Rule

When GitMoneyOS finds a quality gap, it should not only fix the artifact. It should patch the operating system.

Self-healing loop:

1. Name the failure mode.
2. Record evidence.
3. Patch the smallest durable rule, template, checklist, or skill reference.
4. Promote the patch through GitHub.
5. Add a done-when check so future work can catch the same drift earlier.

## Quality Bar For Public Claims

Safe public claim:

> GitMoneyOS has demonstrated a working AI Office control model with live GitHub evidence, public/private repo separation, approval packets, self-audit, and remediation.

Unsafe public claim:

> GitMoneyOS guarantees investment readiness, acquisition readiness, legal defensibility, or security compliance.
