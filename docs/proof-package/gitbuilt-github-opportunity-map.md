# GitBuilt × GitHub Opportunity Map

## Purpose

Frame a future GitHub sponsorship conversation around measurable builder activation, reviewable proof, responsible agentic experimentation, and ecosystem contribution.

GitBuilt is independent and is not sponsored, endorsed, or affiliated with GitHub unless a written agreement says otherwise. Every proposed GitHub contribution below is an idea for a future conversation, not a confirmed benefit or commitment.

## Audience

- GitMoney and GitBuilt owners
- prospective program and curriculum partners
- GitHub ecosystem, education, social-impact, and workforce-development stakeholders
- sponsors evaluating an evidence-based builder pilot

## The Opportunity

GitBuilt can become more than a course about using GitHub. It can operate as a small, governed workforce-development lab that turns learners into builders with inspectable evidence.

The public thesis is:

> GitMoney governs the work; GitBuilt transforms the builder; GitHub records the work; BuildBoard proves the outcome.

The proposed model has two tiers:

1. **Foundations-to-Proof:** an accessible cohort experience in which each participant completes a GitHub-native lesson and produces one reviewable proof artifact.
2. **Agentic Proof Lab:** a smaller advanced cohort that tests guarded agentic workflows and produces one publishable Action or MCP asset with human-review and security evidence.

## Opportunity Lanes

### 1. GitHub Skills Foundry

Convert a core GitBuilt lesson into a short, repository-based learning experience with automated feedback. A working course is stronger sponsorship evidence than a slide deck because a partner can inspect the learner journey, completion path, and proof receipt.

Evidence path: [GitHub Skills quickstart](https://skills.github.com/quickstart).

### 2. Certification-to-Proof

Map role-relevant GitHub certifications to GitBuilt learning paths, while keeping certification optional where it does not improve a participant's goals. Every certification pathway should end in a portfolio or contribution artifact rather than a credential alone.

Evidence path: [GitHub Certifications](https://learn.github.com/certifications).

### 3. Agentic Proof Lab

Use current GitHub agentic surfaces for a bounded advanced lab: Agentic Workflows, portable agent skills, the Copilot SDK, and MCP. Require human approval, scoped permissions, provenance, scans, and intervention records.

Evidence paths: [Agentic Workflows](https://github.blog/changelog/2026-02-13-github-agentic-workflows-are-now-in-technical-preview/), [agent skills with GitHub CLI](https://github.blog/changelog/2026-04-16-manage-agent-skills-with-github-cli/), and [Copilot SDK](https://github.blog/changelog/2026-06-02-copilot-sdk-is-now-generally-available/).

### 4. BuildBoard As Ecosystem Infrastructure

Prove the BuildBoard rubric manually, then evaluate whether the repeatable workflow belongs in a GitHub Action, GitHub App, or MCP tool. Only pursue Marketplace or Technology Partner routes after users, documentation, support ownership, and evidence exist.

Evidence paths: [GitHub Marketplace for apps](https://docs.github.com/en/apps/github-marketplace/github-marketplace-overview/about-github-marketplace-for-apps) and [GitHub Technology Partners](https://github.com/partners/technology-partners).

### 5. GitMoney Implementation Practice

Use GitMoney as the implementation and governance layer behind GitBuilt. Once certifications and case evidence exist, evaluate the Services & Channel Partner path for AI Office setup, workflow governance, enablement, and adoption support.

Evidence path: [GitHub Services & Channel Partners](https://github.com/partners/service-channel-partners).

### 6. Proof for Good

Give participants a structured route to contribute to open-source or social-impact projects. This tests collaboration, documentation, contribution quality, and stewardship in a real public environment.

Evidence path: [GitHub for Good First Issue](https://socialimpact.github.com/insights/for-good-first-issue/).

### 7. Workforce Intelligence And Career Literacy

Use the [GitHub Innovation Graph](https://innovationgraph.github.com/) as macro context for curriculum research, and connect proof packages to both technical and nontechnical career families. This is career literacy, not a promise of interviews, certification, or employment.

Evidence path: [GitHub careers for experienced professionals](https://www.github.careers/experienced-professionals).

### 8. GitHub Spark Bridge

Use [GitHub Spark](https://github.com/features/spark) as a rapid-prototyping lane for designers, marketers, operators, and other builders who may not begin with traditional software development. Access and eligibility must be confirmed before a pilot promises seats or usage.

## Proposed Pilot

### Foundations-to-Proof

- ten participants
- one interactive GitHub-native course
- one reviewable contribution or proof artifact per participant
- a manual BuildBoard scorecard
- optional role-relevant certification readiness
- 30- and 60-day outcome follow-up

### Agentic Proof Lab

- a smaller advanced group
- one guarded agentic workflow
- one publishable Action or MCP asset
- a Secure Build Passport recording scans, provenance, cost or usage, human review, interventions, and fixes

## Future Sponsorship Inventory

Subject to eligibility, availability, and written confirmation, a future proposal could ask GitHub for:

- Copilot or Spark seats, or capped AI usage credits
- nonduplicative certification vouchers
- one expert or product-feedback session
- technical review and appropriate ecosystem routing
- later case-study or amplification consideration after outcomes are validated

GitBuilt would provide:

- a governed cohort and learning environment
- a working GitHub-native course
- reviewable participant artifacts
- public-safe activation, completion, intervention, and support-cost evidence
- a de-identified product-feedback memo
- one advanced ecosystem-ready artifact

The first sponsorship request should reduce access friction and improve the pilot. A larger cash or amplification conversation should follow evidence, not precede it.

## Evidence Standard

Track invited, activated, and completed participants; time to first accepted artifact; contribution completion; human interventions; support time; and 30- and 60-day portfolio, contribution, employment, client, or product outcomes.

[Repository-level Copilot usage metrics](https://github.blog/changelog/2026-07-17-repository-level-github-copilot-usage-metrics-generally-available) may strengthen measurement for eligible organizations with the required permissions. Access must be verified before those metrics become a pilot dependency.

## Lifecycle Guardrails

- Do not build a GitHub Models lesson; GitHub announced [full retirement on July 30, 2026](https://github.blog/changelog/2026-07-01-github-models-is-being-fully-retired-on-july-30-2026/).
- Do not build on GitHub App-based Copilot Extensions; that path was [deprecated and retired](https://github.blog/changelog/2025-09-24-deprecate-github-copilot-extensions-github-apps/). Use current MCP, SDK, or custom-agent paths.
- Do not promise Marketplace, partner, registry, accelerator, fund, certification, employment, amplification, or hiring acceptance.
- Recheck previews, program windows, product availability, and participant eligibility before delivery or outreach.
- Productize BuildBoard only after the manual rubric proves repeatable.
