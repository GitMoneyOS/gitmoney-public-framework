# Three-Layer Glass House

## Purpose

Explain the GitMoneyOS operating model with enough transparency that founders can inspect the method, without exposing the private doctrine that makes the system proprietary.

Plain-English translation: GitMoneyOS is a glass house around the workflow, not an open vault for every private thought.

## The Three Layers

| Layer | Founder Name | Public Visibility | Private Boundary | Proof |
| --- | --- | --- | --- | --- |
| Layer 1 | Source Memory | We can show the rules for how memory gets promoted. | We do not publish raw founder doctrine, private strategy, or unapproved notes. | A note or source signal has owner, status, and promotion target. |
| Layer 2 | Agent Workbench | We can show the brief, boundaries, and expected evidence. | We do not expose private prompts, credentials, client context, or hidden decisions. | An agent task has scope, forbidden actions, output, approval path, and evidence. |
| Layer 3 | GitHub Record | We can show issues, PRs, commits, labels, docs, and public proof. | Private repos stay private when they hold source doctrine, client delivery, or internal operations. | A GitHub issue, PR, commit, review, setting, or governance record exists. |

## Founder Translation

- Source Memory is where thinking gets shaped.
- Agent Workbench is where bounded work gets produced.
- GitHub Record is where approved work becomes inspectable.

The system works when the founder can tell which layer something belongs to and what proof moves it forward.

## What We Show Publicly

GitMoneyOS publicly shows:

- the three-layer model
- the public repo
- public issues and PRs
- proof package
- founder demo
- self-audit summary
- quality standard
- claim boundaries

This is enough for a founder to understand the operating system without seeing private operating doctrine.

## What We Keep Private

GitMoneyOS keeps private:

- master source document
- raw strategy notes
- unapproved doctrine
- private prompts
- client delivery material
- internal pricing
- credentials or access details
- sensitive partner context

Private does not mean vague. It means governed.

## The Glass House Principle

Show the workflow. Protect the source.

That means:

- public enough to prove the method
- private enough to protect the proprietary source
- structured enough that a founder can inspect the claim
- honest enough to show gaps and next actions

## Layer Movement

GitMoneyOS does not treat every idea as official truth.

Work moves like this:

1. A source idea, risk, or note starts in private memory.
2. An agent or human turns it into a bounded artifact.
3. The artifact becomes a GitHub issue, PR, doc, or governance record.
4. The official record stores what changed, who approved it, and why.

If the work skips a layer, risk increases.

## Presentation Checkpoint

A founder should be able to say:

> I can see the operating model, the proof trail, and the quality bar. I do not need access to the private vault to understand why the system works.

## Quality Boundary

Safe public proof:

- issue-to-PR workflow
- self-audit summary
- proof package
- public demo
- high-level repo map
- quality and remediation rules

Protected private material:

- source doctrine
- client data
- private strategy
- internal prompts
- sensitive implementation details

## Why This Matters

A founder does not need blind trust or total exposure.

They need inspectable evidence that the operating system can protect knowledge, govern AI-assisted work, and improve itself.
