# Onboarding Curriculum Path

## Purpose

Show how a founder moves from "GitHub feels technical" to "I can govern my AI Office."

## Owner

Red Pillar and Hitsuyo Aku, under KnowTheLedge.

This is the curriculum path behind the public proof package.

## Learning Standard

Each step uses:

- one plain-English idea
- one visible example
- one small action
- one proof checkpoint

## Level 1: Understand The AI Office

Business job:

- Give the founder a mental model for why GitHub matters.

Teach:

- GitHub is the official company record.
- Private memory is not the official record.
- Agent output is not approved truth.

Proof:

- Founder can explain the three layers: memory, workbench, record.

## Level 2: Read The Public Surface

Business job:

- Build confidence without asking the founder to edit anything.

Teach:

- README
- issue
- pull request
- commit

Proof:

- Founder can find the public proof package, one issue, and one PR.

## Level 3: Turn A Messy Thought Into An Issue

Business job:

- Convert scattered work into a trackable business task.

Teach:

- issue title
- why it matters
- owner
- done-when checklist
- evidence

Proof:

- Founder can draft one issue from a messy note.

## Level 4: Understand The Approval Packet

Business job:

- Prevent important changes from becoming official without review.

Teach:

- branch as draft room
- PR as approval packet
- merge as official record

Proof:

- Founder can inspect a PR and say what decision is being made.

## Level 5: Score The AI Office

Business job:

- Show what evidence exists and what remains risky.

Teach:

- Exit-Ready Score
- observed evidence
- inferred judgment
- unknowns

Proof:

- Founder can name top three risks and next three actions.

## Level 6: Govern Monthly

Business job:

- Keep the system from drifting after setup.

Teach:

- access review
- stale issues
- stale PRs
- security alerts
- agent-rule changes

Proof:

- Founder can run or review a monthly governor issue.

## Accelerator Output

By the end, the founder should have:

- current operating map
- top three scattered assets
- top three risks
- recommended repo map
- first issue draft
- first PR-ready approval packet
- first governor review checklist

## Accessibility Standard

The core path should work for a founder who:

- has never used GitHub
- is skimming between calls
- uses keyboard navigation
- uses a screen reader
- needs the business reason before the technical word

Each module needs visible screen checkpoints and no color-only meaning.
